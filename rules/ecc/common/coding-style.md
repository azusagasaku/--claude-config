# 编码风格

## 不可变性（CRITICAL）

始终创建新对象，禁止修改现有对象：

```
// 伪代码
错误:  modify(original, field, value) → 原地修改原对象
正确:  update(original, field, value) → 返回新副本，原对象不变
```

理由：不可变数据消除隐藏副作用，降低调试难度，保障并发安全。

## 核心原则

### KISS（保持简洁）

- 优先选择能正常运行的最简方案
- 避免过早优化
- 清晰度优先于技巧性

### DRY（避免重复）

- 重复逻辑提取为共享函数或工具
- 避免复制粘贴导致的实现分化
- 仅在重复确实存在时才抽象，不依赖猜测

### YAGNI（你不需要它）

- 不提前构建尚未需要的功能或抽象
- 避免猜测性的通用化设计
- 从简单方案起步，确实需要时再重构

## 文件组织

多小文件优于少量大文件：
- 高内聚、低耦合
- 典型范围 200-400 行，上限 800 行
- 大型模块中的工具函数应独立抽取
- 按功能/领域组织，不按文件类型分类

## 错误处理

必须全面处理错误：

- 每一层都显式处理错误
- 面向用户的代码提供友好的错误信息
- 服务端记录详细的错误上下文
- 禁止静默吞掉错误

## 输入验证

系统边界处必须执行验证：

- 所有用户输入先验证再处理
- 优先使用 Schema 验证
- 快速失败并给出明确的错误信息
- 不信任任何外部数据（API 响应、用户输入、文件内容）

## 命名约定

- 变量和函数：`camelCase`，命名具有描述性
- 布尔值：优先使用 `is`、`has`、`should`、`can` 前缀
- 接口、类型和组件：`PascalCase`
- 常量：`UPPER_SNAKE_CASE`
- 自定义 Hook：`camelCase`，以 `use` 为前缀

## 代码坏味道

### 深层嵌套

逻辑嵌套加深时，优先使用 early return 替代深层 if-else。

### 魔法数字

为有意义的阈值、延迟和限制定义命名常量。

### 长函数

将大型函数拆分为职责明确的小函数。

## 代码质量检查清单

任务完成前逐项确认：

- [ ] 代码可读、命名清晰
- [ ] 函数短小（<50 行）
- [ ] 文件聚焦（<800 行）
- [ ] 嵌套不超过 4 层
- [ ] 错误处理完善
- [ ] 无硬编码值（使用常量或配置）
- [ ] 无变异操作（使用不可变模式）
