# 常用模式

## 骨架项目

实现新功能时，按以下步骤操作：
1. 搜索经过实战验证的骨架项目
2. 使用并行 Agent 评估候选方案：
   - 安全性评估
   - 可扩展性分析
   - 需求匹配度
   - 实施方案评估
3. 选择最佳匹配的骨架项目作为基础
4. 在其已验证的结构上继续迭代

## 设计模式

### Repository 模式

将数据访问逻辑封装在统一接口之后：
- 定义标准操作：findAll、findById、create、update、delete
- 具体实现层负责存储细节（数据库、API、文件等）
- 业务逻辑仅依赖抽象接口，与底层存储解耦
- 便于切换数据源，也便于用 mock 编写测试

### API 响应格式

所有 API 响应应使用统一的封装结构：
- 包含成功/失败状态标记
- 数据载荷（出错时置空）
- 独立的错误信息字段（成功时置空）
- 分页响应附带 metadata，包含 total、page、limit
