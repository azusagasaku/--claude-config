# 开发工作流

> 本文延续 [common/git-workflow.md](./git-workflow.md)，描述 git 操作之前的完整功能开发流程。

功能开发遵循以下主线：调研 -> 规划 -> TDD -> 代码审查 -> 提交。

## 功能实现工作流

0. **调研与复用** _（新功能开发前必读）_
   - **第一步 — GitHub 代码搜索：** 首先运行 `gh search repos` 和 `gh search code` 查找现有实现、模板和模式，不要直接从头编写。
   - **第二步 — 查阅文档：** 通过 Context7 或官方文档确认 API 用法、包行为和版本细节后再动手。
   - **前两步无法满足需求时才使用 Exa：** GitHub 搜索和官方文档无法覆盖时，再使用 Exa 做更广泛的搜索。
   - **检查包注册表：** 编写工具代码前，先搜索 npm、PyPI、crates.io 等注册表。优先选用经过验证的第三方库，避免自行实现通用工具。
   - **寻找可适配的实现：** 开源项目中能解决 80% 以上问题的，优先 fork、移植或封装，而不是从零构建。
   - 总结：能用现成的就用现成的，能改就改，非必要不从零开始。

1. **先规划后执行**
   - 使用 **planner** Agent 生成实现计划
   - 编码前先生成以下文档：PRD、架构设计、系统设计、技术文档、任务清单
   - 明确依赖关系和风险点
   - 分解为多个实施阶段

2. **TDD 开发流程**
   - 使用 **tdd-guide** Agent
   - 先写测试，确保失败（RED）
   - 编写实现，使测试通过（GREEN）
   - 重构优化（IMPROVE）
   - 验证覆盖率达到 80% 以上

3. **代码审查**
   - 代码完成后立即使用 **code-reviewer** Agent 审查
   - CRITICAL 和 HIGH 级别问题必须修复
   - MEDIUM 级别问题尽量修复

4. **提交与推送**
   - 编写详细的 commit message
   - 遵循 conventional commits 格式
   - 提交格式与 PR 流程详见 [git-workflow.md](./git-workflow.md)

5. **PR 提交前检查**
   - 确认 CI/CD 全部通过
   - 解决所有合并冲突
   - 确认分支与目标分支同步
   - 上述条件满足后再发起 review
