# 代码审查标准

## 目的

代码审查确保代码在合并前具备足够的质量、安全性和可维护性。本规则定义何时审查以及如何审查。

## 强制审查场景

**以下情况必须执行审查：**

- 代码编写或修改完成后
- 向共享分支提交前
- 修改了安全敏感代码（认证、支付、用户数据）
- 架构变更
- 合并 Pull Request 前

**审查前的准备：**

- CI/CD 全部通过
- 无合并冲突
- 分支已同步目标分支

## 审查检查清单

代码完成前逐项确认：

- [ ] 代码可读、命名合理
- [ ] 函数规模可控（<50 行）
- [ ] 文件内聚（<800 行）
- [ ] 嵌套层级不超过 4 层
- [ ] 错误已显式处理
- [ ] 无硬编码的密钥、密码、token
- [ ] 无 console.log 等调试残留
- [ ] 新功能已编写测试
- [ ] 测试覆盖率达到 80% 以上

## 安全审查触发条件

**遇到以下情况，停止当前操作，调用 security-reviewer Agent：**

- 认证或授权代码
- 用户输入处理
- 数据库查询
- 文件系统操作
- 外部 API 调用
- 加密操作
- 支付或财务代码

## 问题严重等级

| 等级 | 含义 | 处理方式 |
|-------|---------|--------|
| CRITICAL | 安全漏洞或数据丢失风险 | **阻止合并** — 必须修复 |
| HIGH | Bug 或显著质量问题 | **警告** — 建议修复后合并 |
| MEDIUM | 可维护性隐患 | **提示** — 建议修复 |
| LOW | 风格或次要建议 | **建议** — 可选修复 |

## 审查 Agent

| Agent | 职责 |
|-------|---------|
| **code-reviewer** | 通用代码质量、设计模式、最佳实践 |
| **security-reviewer** | 安全漏洞、OWASP Top 10 |
| **typescript-reviewer** | TypeScript/JavaScript 专项 |
| **python-reviewer** | Python 专项 |
| **go-reviewer** | Go 专项 |
| **rust-reviewer** | Rust 专项 |

## 审查流程

```
1. 运行 git diff 了解变更范围
2. 首先检查安全清单
3. 检查代码质量清单
4. 运行相关测试
5. 确认覆盖率 >= 80%
6. 选择合适的 Agent 进行详细审查
```

## 常见问题

### 安全

- 硬编码凭证（API key、密码、token）
- SQL 注入（字符串拼接查询）
- XSS 漏洞（未转义的用户输入）
- 路径遍历（未过滤的文件路径）
- 缺少 CSRF 保护
- 认证绕过

### 代码质量

- 函数过长（>50 行）— 拆分为小函数
- 文件过大（>800 行）— 提取模块
- 嵌套过深（>4 层）— 使用 early return
- 错误处理遗漏 — 必须显式处理
- 大量变异操作 — 优先使用不可变模式
- 缺少测试 — 补齐测试

### 性能

- N+1 查询 — 使用 JOIN 或批量查询
- 缺少分页 — 为查询添加 LIMIT
- 无边界查询 — 添加约束条件
- 缺少缓存 — 为高开销操作添加缓存

## 审批标准

- **批准**：无 CRITICAL 和 HIGH 级别问题
- **警告**：仅存在 HIGH 级别问题（审慎合并）
- **阻止**：存在 CRITICAL 级别问题
